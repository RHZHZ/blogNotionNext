# UI 开发规范

> `UIplan/` 目录用于保留当前阶段的 UI 规划镜像与工作副本，便于集中讨论和串联阶段性方案；正式归档与长期维护仍以项目根目录下的同名文件为准。若两处内容需要同步，应优先更新主目录版本，再按需回写本目录。

## 1. 目标
- 保持界面一致性、可读性、可维护性。
- 优先保证信息层级清晰，再追求视觉效果。
- 主题色只做点缀，不主导大面积背景。

## 2. 设计原则
- 中性色做基础层，强调色做辅助层。
- 一个区域只保留一个视觉重点，避免多个高亮同时竞争。
- 优先提升间距、层级、边界，再考虑高饱和颜色。
- 动效服务于反馈，不制造干扰。

## 3. 色彩规范
### 3.1 基础色
- 页面背景、卡片背景、分组容器优先使用中性色。
- 亮色模式下保持浅底深字；暗色模式下保持深底浅字。
- 禁止在一个面板内同时使用多种高饱和背景色。

### 3.2 强调色
- 强调色默认来自品牌色或封面取色系统。
- 强调色只用于以下场景：
  - 当前态
  - 图标弱底色
  - 边框轻强调
  - 小面积 badge
  - hover / focus 微反馈
- 强调色不应直接铺满整块面板背景。
- 封面取色必须弱化使用，优先通过 `color-mix`、透明度、边框、阴影内光实现。

### 3.3 字体与对比度
- 正文与主要操作文字必须优先保证可读性。
- 亮色模式建议深灰字，避免纯黑生硬。
- 暗色模式建议浅灰到白，不要大面积使用纯白高亮。
- 强调色文字只用于短文本或小面积状态，不用于长段落。

## 4. 层级规范
建议按以下层级组织：
1. 页面层：背景、整体结构
2. 区块层：卡片、容器、分组
3. 内容层：标题、正文、辅助信息
4. 交互层：按钮、链接、标签、菜单项
5. 状态层：当前页、hover、active、disabled、loading

要求：
- 层级越高，变化越克制。
- 同一屏内的卡片样式尽量统一圆角、边框、阴影节奏。

## 5. 间距规范
- 优先使用稳定的 4 / 8 / 12 / 16 / 20 / 24 节奏。
- 同类型组件保持一致内边距。
- 分组与分组之间的间距要明显大于组内元素间距。
- 移动端优先保证点击热区，不要为了紧凑压缩到难以操作。

## 6. 圆角与边框
- 小元素：`10px ~ 14px`
- 按钮 / 标签 / 菜单项：`14px ~ 18px`
- 卡片 / 抽屉区块：`20px ~ 28px`
- 同一模块内圆角等级不超过 2 种。
- 边框以弱对比为主，强调边框仅用于当前态或焦点态。

## 7. 阴影与玻璃感
- 阴影用于分层，不用于“做脏”。
- 亮色模式阴影更柔、更散；暗色模式阴影更浅、更克制。
- 玻璃感必须建立在可读性之上，不能因透明导致文字发灰或背景脏乱。
- 有封面取色联动时，优先把颜色放进边框、内阴影或极轻背景混色中。

## 8. 组件规范
### 8.1 按钮
- 默认态保持中性。
- hover 只做轻微位移、亮度变化或弱缩放。
- 主要按钮与次级按钮必须通过层级区分，不靠多个高饱和色区分。

### 8.2 菜单
- 当前页必须有明确但克制的高亮。
- 分组展开态与当前命中态可叠加，但不能过度发光。
- 图标、文字、箭头应属于同一视觉系统。

### 8.3 标签
- 标签优先使用浅底、弱边框。
- 选中态用强调色弱混合，不用整块纯色硬铺。
- 标签数量多时，要优先考虑换行、截断和触控体验。

### 8.4 卡片
- 卡片背景、边框、阴影必须成套出现。
- 卡片内标题、说明、操作按钮要有清晰主次。
- 一块卡片内不应出现过多独立视觉中心。

## 9. 动效规范
- 动效时长建议：
  - 微交互：`150ms ~ 250ms`
  - 面板开合：`300ms ~ 500ms`
- 优先使用缓动曲线制造自然感，避免机械线性动画。
- 动效组合建议最多包含：位移 + 透明度，必要时再加轻微缩放。
- 避免大位移、弹跳过强、连续闪烁。

## 10. 响应式规范
- 移动端先保证单手操作舒适度。
- 抽屉、弹层、菜单按钮要优先保证点击面积。
- 小屏下优先纵向堆叠，减少横向拥挤。
- 文字过长必须有截断或换行策略。

## 11. 暗色模式规范
- 不直接反转亮色样式，要重新检查层级关系。
- 暗色模式下的边框、阴影、背景混色都要降低存在感。
- 同一个强调色在暗色模式下应更柔和，而不是更刺眼。

## 12. 开发落地建议
- 新组件开发顺序：结构 > 层级 > 间距 > 颜色 > 状态 > 动效。
- 先定义默认态，再补 hover / active / disabled / loading。
- 开发完成后至少检查：
  - 亮色模式
  - 暗色模式
  - 移动端
  - 当前态 / hover 态 / 展开态
  - 长文案 / 空数据 / 多数据

## 14. 顶部导航与文章页标题胶囊补充规范

### 14.1 适用范围
- `themes/heo/components/Header.js`
- `themes/heo/components/MenuListTop.js`
- `themes/heo/components/MenuItemDrop.js`
- `themes/heo/style.js` 中顶部导航相关规则

### 14.2 本轮设计结论
本轮顶部导航最终采用“稳定三栏 + 中央独立胶囊 + 统一交互态”的方案：
- 左侧品牌区、中央导航区、右侧操作区结构解耦，避免右侧阅读进度出现后带动左侧视觉重心变化。
- 中央 `headline` 不再承担视觉容器职责，只负责定位与切换动画。
- 文章页中央文案改为显示当前文章标题，不再显示 `AUTHOR | BIO`。
- 中央标题胶囊采用更强的玻璃拟态 / 磨砂视觉，但高度、圆角、阴影节奏与左右胶囊保持同体系。
- 顶层导航项与下拉菜单项统一 `hover / focus-visible / current` 的视觉语言，并分别适配亮色与暗色模式。

### 14.3 结构规范
- 顶部导航应优先采用三栏结构：品牌区 / 中央区 / 操作区。
- 中央区中的菜单与标题采用切换关系，不应同时抢占视觉中心。
- 胶囊容器职责必须单一：
  - `.heo-top-nav__headline` 负责定位
  - `.heo-top-nav__headline-pill` 负责磨砂、边框、阴影、圆角与内边距

### 14.4 视觉规范
- 顶部胶囊统一使用大圆角 `999px`。
- 文章页标题胶囊高度应与左右胶囊接近，优先保持 `2.7rem ~ 2.9rem` 的最小高度区间，而不是单独做成更厚的主按钮感。
- 磨砂效果优先通过以下组合实现：
  - 半透明中性背景
  - 弱高光边框
  - 柔和外阴影 + 轻内高光
  - `backdrop-filter: blur(...) + saturate(...)`
- 亮色模式下标题文字应明显深于普通正文，但仍需与左右导航文字保持同一深灰体系，避免中心标题因透明字色或过强白色文字阴影而显得发灰。
- 文章页中央标题最终建议采用“深灰主体字色 + 极弱或无高光字阴影”的方案，优先保证清透和识别度，再考虑玻璃感氛围。
- 暗色模式下保持浅暖色或浅灰高对比文字，但不要使用刺眼纯白大面积铺色。

### 14.5 交互规范
- 顶部导航项、下拉触发项、下拉菜单项必须共享同一套状态语言：
  - `hover`：轻上浮、弱混色背景、轻描边、轻阴影
  - `focus-visible`：在 `hover` 基础上增加 ring
  - `current/active`：比默认态高一档，但不能重到破坏整体中性色基底
- 图标、文字、箭头必须跟随 `currentColor` 统一变化。
- 下拉菜单与触发器之间不得存在导致 hover 丢失的断层，需要保留可连续悬停的桥接命中区。

### 14.6 文章页专项约束
- 文章页顶部通常存在更高优先级的 `post + flat` 规则，新增交互态时必须显式检查是否被文章页专属规则覆盖。
- 文章标题过长时，后续应补充桌面端截断或渐隐策略，避免中央胶囊过度挤压左右区。

### 14.7 token 化落地结论
- 顶部导航相关视觉参数应优先收敛到 `themes/heo/style.js` 的 token 层，再由品牌区、标题胶囊、菜单组、下拉面板复用。
- 本轮已收敛的 token 类型至少包括：
  - 导航胶囊容器：圆角、边框、背景、阴影、模糊
  - 标题胶囊：最大宽度、最小高度、内边距、字色、字阴影
  - 文章页标题胶囊：独立磨砂强度、独立文字对比度
  - 菜单交互态：`hover / focus-visible / current`
  - 下拉面板：边框、背景、阴影、模糊
- `Header.js` 应仅保留结构与状态切换逻辑，不再承载顶部导航视觉细节。
- 文章页专属覆盖也应基于同一组 token 做差异化，而不是重新散写一套硬编码值。

### 14.8 下一轮开发与验收建议
1. 做一轮桌面端 / 移动端 / 普通页 / 文章页 / 亮色 / 暗色六维回归检查，确认 token 化后无视觉漂移。
2. 重点检查长标题文章、带下拉菜单导航项、`ReadingProgress` 出现时的右侧操作区节奏是否稳定。
3. 为导航相关 token 进一步区分“容器 token”和“交互 token”，减少语义混用。
4. 若后续继续增强，可补充下拉面板打开动效、长标题 tooltip 体验和截图型验收基线。
5. 完成回归后，再决定是否把这套 token 继续下放到移动端抽屉、搜索面板和浮层类组件。

## 15. 首页卡片与侧栏卡片统一补充规范

### 15.1 当前阶段判断
在顶部导航、文章页标题胶囊、下拉面板和导航交互态完成第一轮统一后，当前 `HEO` 主题的主要割裂点，已经从“导航内部不统一”转为“导航已进入新体系，但页面卡片仍大量保留旧样式语言”。

当前问题主要表现为：
- 顶部导航已具备统一圆角、边框、背景、阴影和交互 token，但首页内容卡片、Hero 卡片、右栏卡片仍存在大量旧式 `bg-white + dark:bg-[#1e1e1e] + border + rounded-xl` 组合。
- `SideRight.js`、`Hero.js`、`BlogPostCard.js` 等组件中仍分散存在内联样式、局部硬编码颜色和独立卡片节奏。
- 首页主卡片、右栏 section 卡片、作者信息卡片之间尚未建立同源视觉语言，导致页面出现“顶部新、正文旧”的割裂感。

### 15.2 适用范围
- `themes/heo/components/Card.js`
- `themes/heo/components/BlogPostCard.js`
- `themes/heo/components/SideRight.js`
- `themes/heo/components/InfoCard.js`
- `themes/heo/components/Hero.js`
- `themes/heo/style.js` 中首页卡片、右栏卡片和 Hero 卡片相关规则

### 15.3 本轮收口目标
下一阶段不再只做单个组件微调，而是优先建立“首页卡片 + 侧栏卡片”的统一基座：
- 统一卡片圆角等级、边框强度、背景层次、阴影节奏与亮暗色映射。
- 让首页主文章卡片、Hero 区块卡片、右栏 section 卡片、信息主卡进入同一视觉系统。
- 优先把硬编码卡片样式收敛到 `themes/heo/style.js` 的 token 层，再由各组件按语义复用。
- 避免继续出现“导航已是新体系，但页面主体仍是旧卡片语言”的状态。

### 15.4 卡片结构规范
- 卡片应优先拆分为：外层容器 / 头部区 / 内容区 / 操作区，而不是把所有视觉和布局职责堆叠在同一层 class 上。
- 通用 `Card.js` 应逐步承担基础卡片壳职责，用于统一内边距、圆角、边框、背景和阴影基线。
- section 容器负责分区，不应抢占内容卡片的视觉中心。
- Hero 卡片可略强于普通列表卡，但不能脱离同一组圆角、边框和阴影节奏单独成立一套语言。

### 15.5 视觉规范
- 首页卡片与侧栏卡片应共享同一组基础卡片 token：
  - 圆角
  - 边框
  - 背景
  - 阴影
  - 模糊 / 内高光（如有）
- 亮色模式下保持浅底深字、柔和阴影；暗色模式下保持深底浅字，并进一步降低边框与阴影存在感。
- 卡片强调色继续只做点缀，不用于整块硬铺背景。
- 同一屏中若同时出现文章卡片、信息卡片、列表卡片，它们可以有主次差异，但不应出现完全不同的卡片语言。

### 15.6 代码落地约定
- 卡片相关视觉参数应优先沉淀到 `themes/heo/style.js`，避免在 `BlogPostCard.js`、`SideRight.js`、`Hero.js` 中继续散写重复值。
- 组件内联样式仅保留真正依赖运行时数据的差异项，不再承载通用卡片视觉基线。
- `Card.js` 优先作为卡片基座收口入口；首页文章卡片与右栏卡片优先复用同一组基础 token，再按语义做轻微差异化。
- 若某张卡片需要保留更强视觉中心，也应基于同一套 token 增量覆盖，而不是重新硬写一套背景、边框、阴影。

### 15.7 下一轮实施顺序
1. 先改 `themes/heo/components/Card.js` 与 `themes/heo/style.js`，建立统一卡片基座与卡片 token。
2. 再改 `themes/heo/components/BlogPostCard.js` 与 `themes/heo/components/SideRight.js`，让首页主卡片与右栏主面板先进入同一体系。
3. 最后收口 `themes/heo/components/InfoCard.js` 与 `themes/heo/components/Hero.js`，完成首页视觉中心与右栏重点卡片统一。

### 15.9 本轮阶段归档（卡片基座 / 边界收口 / InfoCard 明暗适配）
本轮已经完成首页卡片体系第一阶段收口，重点不再是“把所有卡片都做成同一种重磨砂玻璃”，而是在同一视觉体系内建立不同强度层级。

当前已完成的阶段结论如下：
- 已在 `themes/heo/style.js` 中建立通用卡片基座 token，并让 `Card.js` 承担统一卡片壳职责。
- 首页文章卡 `BlogPostCard.js`、右栏 section 卡、`InfoCard.js`、`TouchMeCard.js`、`Hero.js` 主要卡容器已进入同一套卡片语言。
- 首页文章卡与右栏卡片边界已从“过轻”回调半档，当前方向为“边界可感知，但不回到旧式厚描边”。
- `InfoCard.js` 已确认保留为强视觉卡，但其强视觉只来自同体系内的增量覆盖，而不是脱离基座另起一套样式。
- `InfoCard` 已完成亮色 / 暗色双态分离：
  - 亮色模式：`浅蓝玻璃 + 深字`
  - 暗色模式：`深蓝黑玻璃 + 亮字`
- 运行时颜色 `--heo-infocard-bg` 仍保留，但在亮色模式下只作为轻染色参与，不再直接主导整块背景，以避免白字被品牌色压住。

### 15.10 下一阶段重点（Hero 内部旧色块清理）
下一阶段的主要对象从“首页文章卡 / 右栏主卡”转入 `Hero.js` 内部收口，重点不是重做结构，而是继续统一局部视觉语义：
- 清理 `Banner` hover 遮罩中偏旧的整块高饱和色铺底。
- 清理 `GroupMenu` 三个入口仍各自使用独立重色渐变的旧语言。
- 统一推荐小卡、`TodayCard`、Banner 遮罩、更多按钮之间的边框、背景、阴影与 hover 节奏。
- 保持 Hero 仍然是首页视觉中心，但不能脱离卡片基座语言单独形成第四套系统。

### 15.12 Hero 最终稿归档（统一收口 / 去死板化 / 右侧排版复位）
在完成卡片基座统一和 `InfoCard` 明暗适配后，本轮已进一步完成首页 `Hero` 区域的最终收口，目标不是追求更强视觉刺激，而是让首页视觉中心“统一、通透、有呼吸感，但不死板”。

本轮最终结论如下：
- `Hero` 左侧 `Banner`、下方 `GroupMenu`、右侧推荐小卡、`TodayCard` 已建立同源视觉语言，首页首屏不再出现“左一套、右一套”的割裂。
- `Banner` hover 遮罩已从旧式整片高饱和铺底，收口为更轻的半透明彩色参与层，强化方式改为局部光感、标题位移和轻玻璃感，而不是整块实色压屏。
- `Hero` 的“去死板化”主要通过轻不对称、主次差、局部错位和轻动势实现，而不是靠额外堆叠颜色与装饰。
- `TodayCard` 已从普通封面推荐卡收口为与左侧 `Banner` 同气质的 Hero 卡：
  - badge 弱化
  - eyebrow 减重
  - 标题区上提
  - 更多按钮减重并留出安全区
  - 标题断行与按钮占位重新平衡
- 当前 `Hero` 最终方向已确认：保留首页视觉中心地位，但不再脱离卡片基座语言单独形成新系统。

### 15.13 Hero 最终稿专项规范
- `Banner` hover 视觉强化优先来自遮罩透明度、标题层级、局部光斑和箭头位移，不再使用高饱和整片蒙层做主导。
- `GroupMenu` 可保留语义色差，但强度必须低于 `Banner` 与 `TodayCard`，避免底部三入口反客为主。
- `TodayCard` 的标题区必须优先保证阅读节奏：
  - 避免孤行断句
  - 避免按钮压迫标题区
  - 避免底部遮罩过厚导致整体发闷
- `Hero` 内所有增强元素都应遵守同一原则：先保证可读性和排版，再考虑玻璃感、光感和动势。

### 15.14 当前阶段收尾建议
- 本轮 `Hero` 已可视为首页首屏当前定稿版本，后续若继续优化，应优先转向整站其它卡片与局部面板的统一，而不是继续高频改动 `Hero` 主结构。
- 若后续再做 `Hero` 调整，应限制为毫米级微调，例如：标题断行、局部留白、弱状态颜色和 hover 节奏，不再重开新的视觉方向。

### 15.15 第二轮卡片统一归档（NoticeBar / TouchMeCard / 浮动按钮组 / MenuGroupCard）
在 `Hero` 定稿后，本轮继续推进了首页与常驻入口层的统一收口，目标是把仍然保留旧式 Tailwind 直写或独立皮肤的组件，纳入当前 `HEO` 卡片体系和入口按钮体系。

本轮已完成的阶段结论如下：
- `NoticeBar` 已从旧式横向通知条收口为首页卡片体系内的横幅变体，补齐了 label、正文区、箭头图标的统一语义与亮暗模式状态。
- `TouchMeCard` 保留翻转交互，但其视觉语言已从高饱和活动卡收回到“个性卡但同体系”的实现方式，前后两面都接入了当前卡片标题、eyebrow、CTA 与弱染色逻辑。
- 常驻浮动入口 `FloatDarkModeButton`、`JumpToTopButton`、`JumpToCommentButton` 已统一为同一代 `heo-float-widget-btn` 按钮体系，并补齐 `hover / focus-visible / 暗色模式 / 可访问 button 语义`。
- `MenuGroupCard` 已从旧式纯文本列表收口为可复用的 section/card 入口卡，补齐容器、条目、计数 badge 的层级关系，并与侧栏卡片体系对齐。
- 当前首页、右栏、常驻入口三类高频界面的视觉语言已进一步靠拢，整站完成度明显高于仅完成 `Hero` 和 `InfoCard` 时的阶段状态。

### 15.16 本轮收口原则补充
- 横幅型组件优先作为“卡片变体”处理，而不是保留独立的一条旧横条皮肤。
- 个性组件可以保留行为差异，如翻转、动态图层、数字 badge，但必须回到统一卡片边界、文字层级和弱染色体系。
- 常驻浮动入口必须优先保证语义化与可访问性，能用 `button` 的地方不再使用可点击 `div`。
- 侧栏中的入口类小组件，应优先收口为“卡片里的列表 / 子入口”而不是继续保留散装文本链接。

### 15.18 首页 Hero 轻量书籍预告补充规范
在首页卡片体系、Hero 最终稿和移动端首屏结构已经基本收口后，首页 `Hero` 的书籍内容继续遵循“轻量预告，不做二次书单页”的原则；但在本轮结构评审后，承载方式已从“附着在原 Hero 内的补充卡片”升级为“桌面端双主区、移动端单主视觉 + 书籍入口”的新首屏模型。

#### 15.18.1 目标定位
- 首页 `Hero` 中的书籍内容只负责建立“阅读是长期标签之一”的认知。
- 它应承担的是 `轻量预告 + 跳转承接`，而不是完整书架浏览。
- 首页书籍模块必须服务首屏主视觉，但现在允许作为桌面端 `Hero` 的独立右侧区域存在，不再只是原 Banner 下方的附属卡片。

#### 15.18.2 新版结构结论
- 桌面端 `Hero` 改为 `左幻灯片 / 右书籍展示` 双区域结构。
- 桌面端比例建议优先按 `7:3` 落地；若后续验证阅读权重仍偏弱，再评估微调到 `6.5:3.5`。
- 原 `Hero` 左下三枚分类卡不再作为 Hero 主结构的一部分，建议移出 Hero 主视觉区。
- 原 `Hero` 右侧 `推荐文章组 + TodayCard` 不再保留，避免与书籍区重复竞争首屏焦点。

#### 15.18.3 桌面端承载方式
- 左侧主区负责文章幻灯片或精选文章轮播，承担首页首屏的内容主入口职责。
- 右侧书籍区负责展示 `最近在读` 的轻量阅读预告，建议展示 `2~3` 本书。
- 右侧书籍区应包含：模块标题、少量书封、必要时的书名/作者、以及明确跳转 `我的书单` 的入口。
- 右侧书籍区定位为“次主区”，不能弱到沦为挂件，也不能重到压过左侧幻灯片主视觉。

#### 15.18.4 移动端承载方式
- 移动端不机械复刻桌面双栏，而采用 `上方主幻灯片 + 下方轻量阅读入口` 的纵向结构。
- 移动端阅读区不再展开完整阅读卡信息，统一收敛为 `书籍封面级入口`。
- 移动端建议仅展示 `1~2` 本书封，并保留一个轻量 `我的书单` 跳转按钮。
- 移动端阅读区默认不展示书名、作者、摘要和统计，避免形成第二个重信息区。
- 首页 `CategoryBar` 继续放在 Hero 之后承接，不再与移动端 Hero 主视觉混排。

#### 15.18.5 信息密度约束
- 桌面端书封数量建议 `2~3` 本，移动端建议 `1~2` 本。
- 桌面端可保留极少量辅助信息；移动端原则上只保留封面与跳转，不保留长文案。
- 不允许在 Hero 书籍区中加入长摘要、复杂统计、多分组切换或完整书单浏览交互。
- 不允许让右侧书籍区在视觉上强过桌面端左侧幻灯片主区。

#### 15.18.6 视觉与布局约束
- Hero 书籍模块应继续复用统一圆角、边框、背景、阴影与亮暗色逻辑，但允许作为独立区域形成稳定版面比例。
- 桌面端左 / 右双区应形成清晰主次：左侧内容主视觉，右侧阅读标签展示。
- 移动端优先保证首屏单主视觉路径，阅读模块只能作为主视觉下方的轻入口，不再构成第二主屏。
- 书封可以建立记忆点，但必须控制面积和数量，避免重新把 Hero 变成第二个书单页。

#### 15.18.7 数据与承接约束
- 首页 Hero 书籍模块优先复用 `About / Booklist` 现有阅读数据链路，不单独新增维护源。
- 跳转优先指向 `/booklist`，必要时再补 `About` 页阅读区作为次级承接。
- `Hero` 负责“预告”，`About` 负责“概览”，`Booklist` 负责“展开浏览”，三者职责不得回流混用。

#### 15.18.8 后续实施顺序
1. 先重构 `Hero` 结构，移除旧的 `分类卡 + 推荐文章组 + TodayCard` 主布局。
2. 再按 `7:3` 落地桌面端 `左幻灯片 / 右书籍展示` 新骨架。
3. 再实现移动端 `主幻灯片 + 书封级阅读入口 + Hero 外 CategoryBar`。
4. 再接入现有阅读数据链路与 `/booklist` 跳转入口。
5. 最后做桌面端、移动端、暗色模式和首屏信息密度验收。

## 16. 移动端抽屉与面板类组件补充规范

> 本节已正式接管原 `HEO_MOBILE_PANEL_UI_SPEC.md` 的设计规范内容，作为 `HEO` 移动端面板、抽屉、目录、搜索弹层与小屏阅读态相关规则的唯一维护入口。后续新增结论、归档与验收标准，统一更新到本节，避免多份文档并行导致漂移。

### 16.1 适用范围
- `themes/heo/components/SlideOver.js`
- `themes/heo/components/SideBarDrawer.js`
- `themes/heo/components/Catalog.js` 的抽屉 / 面板变体
- `themes/heo/components/MenuListSide.js`
- `themes/heo/components/MenuItemCollapse.js`
- `themes/heo/components/TagGroups.js` 在抽屉 / 面板中的使用场景

### 16.2 本轮设计结论
本轮移动端抽屉最终采用“顶部品牌卡 + 单层功能入口 + 分区标题 + 托盘容器 + 内容项”的结构，不再允许同一功能块同时出现多层空容器、独立皮肤按钮和无层级关系的子项堆叠。

### 16.3 结构规范
- 抽屉顶部优先拆分为：品牌区 / 关闭操作区。
- 单个功能入口（如暗黑模式切换）优先使用单层功能卡，不再额外包一层空托盘。
- 分区内容优先采用：
  1. 区块标题
  2. 外层托盘
  3. 托盘内内容项
- 快捷入口、标签区、导航区应属于同一层级体系，只允许内容密度不同，不允许容器语言不同。

### 16.4 菜单展开态规范
- 父菜单项与子菜单项必须形成明确承接关系。
- 展开区域只能表现为“父项下挂的一组次级菜单项”，不能做成说明面板、输入框组或大面积留白区。
- 缩进线只作为辅助引导，强度必须低于文字、卡片边界和当前态反馈。
- 子菜单项默认态应可点击、可识别，不能因边框或底色处理不当而显得像禁用态。

### 16.5 视觉规范
- 抽屉面板优先使用中性浅底 / 深底，不使用高饱和整块背景。
- 强调色仅用于：
  - 箭头
  - 图标弱底色
  - 当前态边框轻强调
  - focus-visible / hover 微反馈
- 托盘容器应弱于内容项，不得反客为主。
- 内容项若已经是卡片，不再额外叠加第二层同强度卡片壳。

### 16.6 移动端首页首屏与 Loading Cover 补充归档

#### 16.6.1 当前阶段判断
本轮 UI 开发确认，当前 `HEO` 主题在移动端首页与全局加载态上，已经进入“从局部好看转向整体验收”的阶段。问题不再只是某个组件是否足够显眼，而是：
- 移动端首页首屏信息是否过载。
- 首页 `Hero`、分类入口、文章列表之间是否存在重复表达。
- 暗色模式下当前态是否仍然可靠可识别。
- Loading 遮罩是否同时具备品牌识别、状态提示与足够克制的动效节奏。

本轮的核心方向已收敛为两条主线：
1. 移动端首页首屏优先做减法，保留真正承担首屏视觉中心的部分。
2. Loading Cover 优先建立“品牌文案 + 加载状态文案”的双层级，而不是让所有文字都处于同一视觉强度。

#### 16.6.2 适用范围
- `themes/heo/components/Hero.js`
- `themes/heo/components/CategoryBar.js`
- `components/LoadingCover.js`
- `themes/heo/style.js`

#### 16.6.3 移动端首页首屏结构结论
本轮已确认，移动端首页首屏不再继续沿用“幻灯片区 + 杂志流 + Hero 内分类入口”并存的方案，而是收敛为更清晰的单主视觉结构：
- 顶部导航 / 搜索
- 移动端 `Hero` 幻灯片区
- 首页 `CategoryBar`
- 文章列表

结论说明：
- `Hero` 在移动端只保留最强的一层内容，即横向滑动的精选幻灯片。
- `Hero` 内不再重复承载分类入口，因为首页本身已有 `CategoryBar`。
- 移动端首屏的目标不再是“尽量塞满入口”，而是“先建立视觉中心，再交给下方内容区承接”。

#### 16.6.4 移动端 Hero 规范补充
本轮 `themes/heo/components/Hero.js` 已补入独立的 `MobileHero` 分支，并与桌面端 `Hero` 结构解耦。后续统一遵循：
- 移动端 `Hero` 与桌面端 `Hero` 允许采用不同的信息组织方式，不要求机械复刻。
- 小屏首页首屏优先保证：封面图识别、标题可读、单手滑动顺畅。
- 移动端幻灯片卡片允许更强封面氛围，但不能再叠加第二层重信息流区块。
- 标题、摘要、箭头、badge 只保留一条主阅读路径，避免同一张卡内出现多个争抢焦点的元素。

本轮已确认的实现方向：
- 幻灯片卡片宽度、圆角、高度已做收窄与减重，避免小屏首屏显得过厚过重。
- 幻灯片标题字号与元信息节奏已同步收口，目标是让第一屏更紧凑但仍保留 Hero 感。
- 后续若继续优化 `MobileHero`，应优先做毫米级微调，例如标题断行、卡片间距、摘要长度，而不是重新增加第二套内容区。

#### 16.6.5 CategoryBar 暗色模式补充规范
本轮已确认首页分类栏在亮色模式下表现正常，但暗色模式中当前项高亮背景一度丢失。由此补充约束如下：
- `CategoryBar` 的当前态必须在亮色 / 暗色模式下都具备明确、稳定、可立即识别的高亮反馈。
- 暗色模式不能只保留文字色变化，当前态仍应保留边框、背景与阴影层级。
- 当前项样式在暗色模式下允许更暖、更柔和，但不能因覆盖顺序问题退化为普通标签态。

本轮归档结论：
- `themes/heo/style.js` 中已为 `#category-bar .category-bar-item.selected` 增补暗色模式专属覆盖。
- 后续凡涉及 `selected / current / active` 的首页入口类组件，必须同步验收亮色与暗色双态，避免只在亮色模式下完成视觉闭环。

#### 16.6.6 Loading Cover 结构结论
本轮 Loading Cover 的设计方向已从“单一加载文案 + SVG 动画”收敛为“人物线稿 + 品牌主文案 + 状态副文案”的三层结构：
- SVG 线稿承担视觉识别与等待氛围。
- `loading-text` 承担品牌表达，例如 `RHZ BLOG`。
- `loading-text2` 承担状态表达，例如 `LOADING...`。

后续统一遵循：
- 品牌文案与状态文案不能使用同一种视觉强度。
- 主文案负责品牌感，副文案负责状态感与节奏感。
- Loading 动效的目的应是“让等待有信息与情绪”，而不是制造持续干扰。

#### 16.6.7 Loading 文案层级与动效规范
本轮已正式确立 Loading 文案的双层级规则：
- `loading-text`：品牌主文案，允许保留较强识别度与 glitch 气质。
- `loading-text2`：加载状态副文案，应更轻、更窄、更像系统提示。

本轮已落地的状态文案方向：
- `loading-text2` 使用等宽字体语义，强化“终端 / 打字机 / 打印中”感受。
- 通过 `overflow + width + steps()` 建立逐字打印效果。
- 通过闪烁 caret 建立持续加载中的状态认知。
- 明暗模式分别适配品牌蓝 / 暖金状态色，但整体强度必须低于主品牌文案。

约束补充：
- 副文案不得继续复用 glitch 分身效果，避免与品牌主文案竞争视觉中心。
- 若后续继续增强 `loading-text2`，优先增加极轻系统感或节奏感，不再叠加重故障、强抖动或高频闪烁。
- 打字机效果应优先服务“状态被看见”，而不是追求炫技式复杂动画。

#### 16.6.8 对后续迭代的约束
- 移动端首页首屏后续若继续调整，优先检查信息密度、触控体验与分类承接，不再重新引入重复入口。
- `Hero`、`CategoryBar`、文章列表三者必须维持明确主次：`Hero` 做视觉引导，`CategoryBar` 做筛选承接，列表做内容消费。
- Loading Cover 后续若继续优化人物线稿，应优先提高“这是一个人”的识别度，再考虑角色气质细节。
- 任何 Loading 视觉增强都不得破坏主文案 / 副文案的层级分工。

#### 16.6.9 验收补充
1. 小屏首页首屏是否已从多模块竞争，收敛为单主视觉路径。
2. `MobileHero` 是否仍然具备 Hero 气质，但不再压缩分类栏与正文入口空间。
3. `CategoryBar` 当前项在亮色 / 暗色模式下是否都能被立即识别。
4. Loading 遮罩中，品牌文案与状态文案是否已形成清晰主次，而不是两个标题并列抢戏。
5. `loading-text2` 的打字机效果是否可感知，但不会因为过度闪烁影响整体质感。

### 16.7 移动端搜索弹层专项归档（AlgoliaSearchModal）

#### 16.7.1 问题现象
本轮已确认移动端顶部搜索入口在启用 `Algolia` 时，实际走的是 `components/AlgoliaSearchModal.js` 的弹层实现，而不是 `/search` 页面内搜索布局。

问题表现为：
- 页面位于顶部时，搜索弹层显示基本正常。
- 页面下滑后再打开搜索弹层，面板会出现横向错位、右侧留白、整体像被挤压到局部容器中的现象。

#### 16.7.2 根因结论
本轮确认根因不在搜索输入框本身，也不在搜索结果内容，而在于**弹层挂载位置与祖先层视觉属性冲突**：
- `AlgoliaSearchModal` 原先挂在 `themes/heo/components/SearchButton.js` 内部渲染。
- `SearchButton` 位于 `Header.js` 的移动端顶部操作组中。
- 顶部操作组及导航胶囊容器存在 `backdrop-filter` / 磨砂视觉。
- 页面下滑后，顶部导航会切换为 `fixed` 状态。
- 在此结构下，搜索弹层的 `position: fixed` 在移动端会出现参照系异常，导致弹层不再稳定相对 viewport 定位。

本轮同步确认：
- **面板型弹层不应继续渲染在带 `backdrop-filter`、`filter`、`transform` 或其他可能创建 containing block 的祖先容器内部。**
- **顶部按钮组件只负责触发，不应同时承担全局浮层的挂载职责。**

#### 16.7.3 结构修复方案
本轮对 `components/AlgoliaSearchModal.js` 采用以下修复策略：
- 通过 `createPortal` 将搜索弹层挂载到 `document.body`。
- 让弹层脱离顶部导航、移动端操作组和胶囊容器的祖先层影响。
- 外层容器统一改为 `fixed inset-0` 语义，不再继续依赖 `w-screen / h-screen + 局部 flex 居中` 的旧写法。

结论：
- 对于全局搜索、全局菜单、全局说明层、快捷操作层等全局浮层，优先采用 `portal -> body` 挂载，而不是就地渲染在触发按钮旁边。

#### 16.7.4 布局与视觉收口结论
在完成 portal 化后，本轮同步对移动端搜索弹层做了轻量收口：
- 外层遮罩改为更稳定的全屏覆盖 + 柔和 blur，而不是依赖原局部容器视觉叠加。
- 面板容器改为更稳定的移动端内边距、滚动与最大高度策略。
- 结果列表改为独立滚动区，避免整体面板高度不稳时内容区被异常挤压。
- 面板圆角、边框、背景透明度、阴影节奏向当前 `HEO` 抽屉 / 面板体系靠拢。
- 关闭按钮改为独立圆形按钮，纳入当前面板类组件的操作区语言。
- 手机端底部快捷键提示做减重处理，小屏下不再堆叠完整快捷键说明，默认改为更轻的引导文案。

#### 16.7.5 对后续面板类组件的约束
后续新增或改造移动端面板、搜索层、说明层、快捷层时，统一遵循：
- 全局浮层优先 `portal` 挂载到 `body`。
- 触发器组件不再直接承载浮层 DOM 结构。
- 凡祖先层已使用 `backdrop-filter`、`filter`、`transform`、复杂动画或玻璃容器语义时，必须额外警惕 `fixed` 参照异常。
- 小屏优先保证布局稳定与可读性，再考虑玻璃感和装饰层。
- 搜索面板类组件在移动端优先先保证：定位正确、滚动稳定、关闭路径清晰；视觉增强应放在这些基线之后。

#### 16.7.7 小屏文章页阅读态导航收缩补充归档
在完成顶部导航三栏化、标题胶囊统一、移动端搜索弹层 portal 化之后，本轮进一步确认：**文章页小屏阅读场景需要独立于普通移动导航的“阅读态”规则**，不能继续沿用“左右双胶囊完整常驻”的策略。

本轮结论如下：
- 在小屏文章页中，顶部导航的首要任务应从“完整暴露导航能力”切换为“尽量减少对正文阅读的持续打扰”。
- 当用户进入正文并下滑后，顶部导航不应继续维持完整品牌胶囊 + 完整操作胶囊的双主视觉状态。
- 阅读态应优先收口次级能力，再收口视觉体积，最后才考虑进一步增强动效或材质。

本轮已落地的阅读态规则：
- 仅在**小屏 + 文章页 + 下滑超过阈值**时进入阅读态。
- 阅读态下，左侧品牌胶囊完全隐藏，不再持续占据正文顶部注意力。
- 阅读态下，右侧操作区只保留核心入口：`SearchButton` 与菜单按钮。
- `RandomPostButton` 与 `ReadingProgress` 在该场景中视为次级能力，进入阅读态后默认收起。
- 右侧胶囊在阅读态下同步缩小高度、间距与内边距，以减少纵向占用和视觉压迫。

对后续移动端导航/阅读页的约束补充：
- 小屏阅读态下，导航应优先服务“返回可达、搜索可达、菜单可达”，而不是维持品牌展示完整度。
- 若品牌胶囊与操作胶囊同时存在会压缩正文首屏空间，应优先隐藏品牌胶囊，而不是继续压缩正文内容区。
- 文章阅读页与首页/列表页应允许采用不同的移动端导航显隐策略，避免把“浏览态”和“阅读态”混为一套规则。
- 后续若继续增强该模式，应优先做“滚动方向驱动的恢复/收起节奏”微调，而不是重新增加新的常驻按钮。

验收补充：
1. 小屏文章页顶部时，完整导航是否仍可正常显示。
2. 小屏文章页下滑超过阈值后，左侧品牌胶囊是否完全退出。
3. 阅读态下右侧是否仅保留 `搜索 + 菜单`，并且点击热区足够。
4. 阅读态与非阅读态切换时，顶部整体节奏是否自然，没有明显跳变或抖动。
5. 阅读态下正文首屏可见高度是否明显提升，注意力是否更集中在内容本身。

### 16.8 文章页阅读系统与互动编排补充规范

#### 16.8.1 当前阶段判断
本轮继续梳理 `HEO` 文章页后确认：顶部导航、小屏阅读态、文章头部与页脚区域已经有较明确的 UI 基线，但**真正决定文章页长期体验的一整套“阅读系统”仍缺少统一规划**。目前已有能力主要分散在 `themes/heo/index.js`、`components/NotionPage.js`、`components/Comment.js`、`components/InlineIslandAudio.js` 等实现中，功能可用，但页面级编排规则尚未沉淀。

当前缺口主要集中在以下五类：
- 正文阅读系统
- 评论区 UI 规范
- 音频 / 灵动岛在文章页中的页面级编排
- 文章互动系统
- Footer 作为文章收尾系统的统一规则

#### 16.8.2 适用范围
- `themes/heo/index.js`
- `components/NotionPage.js`
- `components/Comment.js`
- `components/InlineIslandAudio.js`
- `themes/heo/components/PostHeader.js`
- `themes/heo/components/FloatTocButton.js`
- `themes/heo/components/Footer.js`
- 以及文章页正文结束后至全站页脚之间的承接区块

#### 16.8.3 正文阅读系统规范
正文区不应只被视为 `NotionRenderer` 的渲染结果，而应被视为文章页的主舞台。后续规划统一遵循：
- 文章页第一视觉中心是正文内容，而不是头图、装饰波浪、浮动入口或右栏模块。
- 正文容器需要建立清晰的阅读宽度、段落间距、标题层级、媒体插入节奏与长内容滚动节奏，不再只依赖 Notion 默认样式自然落地。
- 图库、代码块、引用块、分栏内容、音频块属于正文的“嵌入式阅读组件”，必须服从正文节奏，而不是各自形成独立视觉体系。
- 当页面存在目录、阅读态导航、浮动目录按钮等辅助能力时，它们都应为正文服务，而不是与正文争夺视觉中心。

后续落地优先级建议：
1. 先统一正文容器宽度、段落节奏、标题间距与媒体区留白。
2. 再统一图库、代码、引用、表格、音频卡等嵌入块的边界语言。
3. 最后再处理更细的阅读动效、标题锚点反馈与沉浸感增强。

#### 16.8.3.1 第一轮已落地收口（正文阅读基础层）
在不重写正文结构、不提前细化所有块级皮肤的前提下，本轮已先完成文章正文基础阅读层的第一轮保守收口，重点是把正文容器、媒体块语义和阅读 surface 建立为可持续扩展的统一底座。

本轮已落地内容：
- `components/NotionPage.js` 已新增 `applyArticleReadingEnhancements({ article, width })` 作为正文阅读增强统一入口，将 gallery 判断与媒体块装饰收口为同一调用链。
- 正文根节点 `#notion-article` 已补齐 `data-heo-reading-surface="true"`、`data-heo-article-surface="article"` 与 `heo-article-surface`，用于后续阅读态样式与页面级编排统一识别。
- 图库行在命中 `heo-image-gallery-row` 后，已同步补齐 `data-heo-gallery="true"`，避免 gallery 只停留在布局层而缺少语义标记。
- 图片、gallery、audio、code、quote、callout、bookmark、pdf、embed 已统一纳入 `heo-article-block` / `heo-article-block--*` 语义体系。
- 图片 caption 已统一补齐 `heo-article-caption`，后续正文媒体说明文案可继续沿此语义层扩展。
- `themes/heo/style.js` 已补入正文基础 token：`--heo-article-surface-max-width`、`--heo-article-media-radius`、`--heo-article-media-border`、`--heo-article-media-bg`、`--heo-article-media-shadow` 及其暗色变量。
- 正文容器已正式收口为统一阅读宽度；音频、书签、PDF、callout、embed、code 等嵌入媒体块已具备同源边界语言。

本轮边界说明：
- 本轮只处理正文阅读系统的“基础层统一”，不等于所有正文块都已完成最终视觉设计。
- `styles/notion.css` 仍保持 Notion 基础样式职责，`HEO` 专属阅读体验继续优先收口在 `themes/heo/style.js`。
- 评论区、互动层、音频深度编排与正文收尾系统仍按 `16.8.4` 之后的章节继续推进。

当前验收补充：
1. 正文是否已具备稳定且可复用的统一阅读宽度，而不再只依赖默认内容流宽度自然生成。
2. gallery、audio、code、bookmark、pdf、embed 是否已进入同一语义命名体系，而不是各自分散加样式。
3. 亮色 / 暗色模式下正文嵌入媒体块是否都已进入同一基础边界语言。
4. 本轮改动是否保持“先统一基础层，再逐块细化”的节奏，而未提前把正文系统重新做成一次大重构。

#### 16.8.3.2 文章页布局协同补充归档（AI Summary / 首屏标题 / 小屏承接）
在正文阅读基础层落地后，本轮继续围绕文章页整体阅读编排做页面级收口，重点不是新增模块，而是修正几个已经影响真实阅读体验的布局副作用：`AI Summary` 与正文阅读壳宽度不一致、首屏玻璃卡标题过早换行、小屏下文章头部与正文白卡轻微重叠。

本轮已确认并完成的结论如下：
- `themes/heo/index.js` 已为 `AISummary` 增加 `heo-article-content-width` 包裹层，使 AI 摘要与正文阅读壳共享同一最大阅读宽度语言，不再出现“正文已收窄但摘要仍明显更宽”的割裂。
- `themes/heo/style.js` 已将 `heo-article-content-width` 与 `heo-article-reading-shell` 一并纳入 `--heo-article-surface-max-width` 约束，并统一采用 `margin: 0 auto` 居中，保证文章正文前后的辅助内容与正文 surface 使用同一宽度基线。
- `themes/heo/components/PostHeader.js` 已将首屏信息区明确拆分为 `post-info-head / post-info-title-zone / post-info-meta-zone` 三段式结构，使分类标签、标题、meta 不再互相挤压，也避免标题长度继续直接撑高整张玻璃卡。
- `themes/heo/style.js` 已进一步为标题区补入独立的垂直节奏约束：标题区改为单独承接与居中分配空间，而不是默认贴近上缘；同时桌面端与手机端标题字号均下调半档，以换取更稳定的卡片比例与整体和谐度。
- 小屏下 `post-info-title` 已从固定较窄字符宽度，调整为 `max-width: min(100%, 12ch)`，并保持 `width: 100%`，避免短行标题在玻璃卡中被过早压窄。
- `themes/heo/style.js` 已取消手机端 `.article.heo-post-shell` 的负 `margin-top`，改为 `margin-top: 0`，用于消除文章头部玻璃卡与下方正文白卡之间的轻微重叠。

本轮设计落点：
- AI 摘要属于正文前置辅助层，应与正文共享阅读宽度体系，而不是作为独立全宽营销卡存在。
- 首屏玻璃卡允许有弹性，但标题区不应因容器策略过度保守而提前断行。
- 小屏文章页的首屏与正文承接应优先保证结构稳定，再考虑视觉叠压或装饰性的“贴边感”。

对后续文章页阅读系统的约束补充：
- 以后新增正文前置说明、摘要、提示卡时，优先复用 `heo-article-content-width` 或同级语义，而不是各自单独决定最大宽度。
- 首屏标题区后续若继续微调，应优先检查可用宽度、图标占位与多行截断策略，避免再次通过压缩容器宽度换取“看似规整”的断行。
- 小屏文章页若再出现头图、正文壳、目录入口等模块互相压叠，应优先检查负 margin、sticky 偏移与 section 间距，而不是单独增加局部 padding 硬撑开。

验收补充：
1. `AI Summary` 是否已与正文阅读壳共享同一最大宽度，而不再显得比正文更宽。
2. 首屏标题在桌面与手机端是否都明显减少“未靠近边界就提前换行”的情况。
3. 手机端文章头部玻璃卡与正文白卡之间是否已消除非预期重叠。
4. 上述修正是否仍保持“正文为主、辅助层减重、首屏稳定承接”的阅读主线。
5. 桌面端正文内容与文章卡边框之间的距离是否已从“明显过远”回到更自然的阅读卡比例。
6. 首屏三段式中，标题区上下留白是否已更均匀，且桌面端与手机端标题字号是否更有利于玻璃卡整体比例协调。

#### 16.8.3.4 桌面端专注阅读开关补充归档（desktop focus reading）
在正文阅读基础层、文章页布局协同与小屏阅读态规则逐步明确后，本轮进一步确认：**桌面端文章页也应具备独立于普通浏览态的“专注阅读”能力**，但它不应被做成重型全屏模式，而应是一种仅作用于文章页、以“减少干扰、适度扩展正文、保留必要导航”为目标的轻量阅读切换。

本轮确认的桌面端设计判断如下：
- 桌面端专注阅读的目标不是把文章页做成阅读器或全屏 PDF，而是在保留博客语境的前提下，进一步把视觉中心让回给正文。
- 专注阅读不应破坏现有文章首屏、正文 surface 与全站导航的整体秩序；它属于“当前阅读偏好切换”，而不是新的全站布局模式。
- 该模式仅建议作用于**桌面端文章页**，移动端继续沿用既有“小屏阅读态”规则，不共用同一套交互入口。

建议落地的核心行为：
- 隐藏或收起桌面端右侧干扰型侧栏内容，使正文成为唯一稳定主视觉。
- 正文阅读宽度在现有基础上适度放宽；当前 `--heo-article-surface-max-width: 54rem` 可在专注阅读态下提升至约 `62rem ~ 64rem`，以换取更舒展的桌面端阅读壳，但不建议超过 `68rem`，避免中文行长过长造成回扫疲劳。
- 文章头部信息区保留，但 meta、辅助说明与非必要强调项应同步减弱一层，避免首屏继续与正文争夺注意力。
- `AI Summary`、内联广告、推荐承接等正文前后辅助层在专注阅读态下应优先弱化或延后呈现，而不是与正文同时保持相同视觉权重。
- 目录不建议彻底消失；推荐从“常驻大侧栏”切换为“轻量浮动入口 + 按需展开目录层”，以保留长文导航能力但减少持续干扰。

对开关位置与交互的建议如下：
- 开关应归属于**文章页局部操作区**，优先放在桌面端文章页右上角的阅读操作胶囊 / 工具区，而不是全站设置项或顶部主导航。
- 命名建议优先采用 `专注阅读`，而非更强烈的 `全屏阅读` / `沉浸模式`；前者更符合当前 `HEO` 主题克制、材质化的语气。
- 切换后应提供轻量状态反馈，例如 `已进入专注阅读`，但提示只短暂停留，不形成持续提示条。
- 交互态应优先采用小幅淡出、正文容器平滑扩展、工具位弱高亮等低扰动动效，不做大位移或强转场。

本轮建议的最小可落地版本：
1. 增加桌面端文章页 `专注阅读` 开关。
2. 开启后收起右侧栏 / 次级信息面板。
3. 将正文最大宽度从 `54rem` 提升到约 `62rem`。
4. 保留一个轻量目录入口，不保留常驻重目录栏。
5. 不改变移动端、不做全屏、不完全移除文章首屏。

对后续实现的约束补充：
- 专注阅读态应优先通过文章页状态类或 data-attribute 驱动，并复用既有正文 token（如 `--heo-article-surface-max-width`），避免再散写一套平行样式体系。
- 目录、AI 摘要、评论入口、互动区等辅助层在该模式下应遵循“可达但减重”的原则，而不是简单全部隐藏。
- 该模式的核心价值是“正文更稳、干扰更少、宽度更舒展”，而不是增加新的视觉噱头。

验收补充：
1. 开启专注阅读后，桌面端正文是否成为唯一稳定主视觉。
2. 正文宽度放宽后，是否明显提升舒展感，但未出现中文行长过长导致的阅读疲劳。
3. 目录是否仍然可达，但不再作为重侧栏持续抢占注意力。
4. `AI Summary`、推荐区、广告等辅助层是否已在该模式下回到次级层。
5. 切换交互是否足够轻，不会破坏当前文章页的整体秩序与材质语言。


本轮已确认并完成的结论如下：
- `themes/heo/components/SideRight.js` 已补充 `heo-side-right-sticky` 语义类，用于统一右侧 sticky 容器的顶部偏移和文章页阅读态承接关系。
- `themes/heo/components/Catalog.js` 已补充 `heo-catalog` 根节点语义类，并为目录卡补齐独立的 `heo-toc-card` 语义层与头部结构，使目录区域不再只是“一个列表”，而成为文章阅读辅助层中的明确 section。
- `themes/heo/style.js` 已统一目录卡与文章首屏的视觉语气：顶部渐变氛围、标题区分隔线、亮/暗色的 `active` 与 `hover` 反馈、以及 sticky 场景下的顶部偏移均已纳入同一套规则。
- 当前文章页右侧目录卡已不再是独立的旧式侧栏组件，而是作为“文章首屏阅读辅助卡”进入与头部信息区一致的阅读节奏体系。

本轮设计落点：
- 目录卡需要服务正文与首屏阅读，而不是抢占文章标题、摘要或正文的第一视觉中心。
- 目录卡头部应具备独立 section 语义，但视觉强度必须弱于文章首屏主标题区。
- sticky 容器应优先保证阅读滚动时的稳定承接，而不是通过更强阴影或更重描边制造存在感。
- 目录项的 `active / hover` 反馈应与文章页当前视觉语言一致，强调“阅读定位”而不是“功能菜单感”。

对后续文章页阅读系统的约束补充：
- 右侧目录卡、顶部阅读态导航、浮动目录按钮都属于正文辅助层，后续增强时应继续避免与正文主链形成多中心竞争。
- 若后续继续压缩移动端文章首屏密度，应优先同步检查目录入口、meta 胶囊与正文起始留白的协同关系，而不是单独微调某一个入口。
- 目录卡的视觉增强应优先发生在 section 结构、标题层级和当前态反馈上，不应重新回到大面积强调色铺底或厚重卡片壳路线。

验收补充：
1. 右侧目录卡在文章首屏附近是否已与头部信息区形成同源节奏，而不是割裂的侧栏旧卡片。
2. sticky 状态下目录卡顶部偏移是否稳定，不会与顶部导航或阅读态切换发生碰撞。
3. 目录卡标题区、正文首屏、目录项当前态之间是否具备统一但克制的高亮语气。
4. 亮色 / 暗色模式下目录卡是否都保持明确层级，而无“套娃框”或“过透明无实体”问题。


#### 16.8.3.5 桌面端专注阅读开关开发任务清单
为避免“设计结论已明确、但落地阶段重新发散”的情况，桌面端 `专注阅读` 功能后续开发统一按以下顺序推进，优先遵循“先能用，再做精”的原则。

第一阶段：最小可用版本
1. 增加桌面端文章页专注阅读状态，仅作用于文章页桌面端，不影响移动端；建议通过文章页根节点状态类或 `data-attribute` 驱动。
2. 在文章页右上角阅读操作区增加 `专注阅读` 开关，采用轻量胶囊按钮，不进入全站设置层或顶部主导航。
3. 开启后先收起右侧干扰型侧栏、目录重卡与次级信息面板，使正文重新成为唯一稳定主视觉。

第二阶段：阅读态减重
6. 让 `AI Summary`、正文前置提示层在专注阅读态下回到辅助层，不再与正文争夺首屏视觉中心。
7. 让推荐区、广告、版权声明等正文后置辅助层在专注阅读态下降低存在感，遵循“可达但减重”。
8. 弱化首屏 meta 与次级说明的视觉权重，保证用户进入正文后的注意力更快回落到内容本身。

第三阶段：交互与体验打磨
9. 为切换动作增加轻量状态反馈，如 `已进入专注阅读` / `已退出专注阅读`，提示短暂存在后自动消失。
10. 为侧栏淡出、正文扩宽与按钮状态增加 `150ms ~ 220ms` 的低扰动平滑过渡，不做大位移或强动效。
11. 使用本地存储记住桌面端用户偏好，下次进入文章页时恢复该状态，但不影响移动端行为基线。

推荐优先修改的实现入口：
- `themes/heo/index.js`
- `themes/heo/components/PostHeader.js`
- `themes/heo/components/FloatTocButton.js`
- `themes/heo/style.js`

推荐实现原则：
- 优先复用 `--heo-article-surface-max-width` 等既有正文 token，并在专注阅读态下通过状态类切换变量值，而不是散写新的局部宽度规则。
- 目录、AI 摘要、推荐区等都视为正文辅助层，应根据层级减重，而不是简单粗暴全部删除。
- 功能验收优先看“正文是否更稳、阅读是否更专注、切换是否足够轻”，而不是看视觉变化是否足够大。

阶段验收顺序建议：
1. 先验收开关是否稳定切换、右侧栏是否正确收起、正文宽度放宽后是否确实更舒展。
2. 再验收目录是否仍然可达、AI 摘要 / 推荐区是否已减重、交互切换是否平滑无跳变。
3. 最后再验收长文阅读收益、亮暗模式一致性与不同桌面宽度下的稳定性。

#### 16.8.3.6 正文单图与正文媒体分级补充规范
在完成正文阅读基础层、首屏承接、目录辅助层与桌面端专注阅读规则之后，本轮进一步确认：**桌面端正文中的单张图片已经不只是“图片大小是否好看”的问题，而是正文媒体语义、阅读节奏与页面视觉中心是否被正确组织的问题。**

当前问题判断如下：
- 正文中的普通单图，在部分场景下已经呈现出接近章节主视觉或第二封面的展示势能。
- 当单图默认接近正文阅读壳的满宽展示时，即使技术上没有超出正文容器，也会在心理感知上形成“过满、过重、过像海报”的问题。
- 页头封面已经承担了文章页主视觉职责，正文普通单图不应再次复制同级强度，否则会与标题区、正文文字和章节节奏形成二次竞争。
- 当前正文媒体虽已进入 `heo-article-block` 语义体系，但“普通单图 / gallery / 展示型图片”之间的视觉强度仍需进一步分级，不宜继续默认按同一展示势能处理。

本轮设计结论：
- 正文普通单图的默认角色应是“阅读中的媒体块”或“正文配图”，而不是“章节主视觉图”。
- 桌面端正文单图应优先服从正文阅读宽度、段落节奏与嵌入块层级，不再以“尽量大、尽量显眼”为默认目标。
- gallery 可以保留比单图稍强的展示感，但仍属于正文嵌入式阅读组件，不能脱离正文节奏单独成立一套海报式视觉系统。
- 若后续存在作品展示图、摄影图、海报图等明确需要更强展示性的内容，应通过独立语义或独立状态处理，而不是默认把所有单图都做大。

#### 16.8.3.6.1 正文普通单图清单
后续所有桌面端正文普通单图，统一按以下清单自检：

1. 语义定位
- 默认按“正文配图 / 说明图 / 插图”处理，而不是按封面图、章节主视觉图处理。
- 如果没有明确业务语义证明该图需要承担展示主角职责，就不应使用高势能展示方案。

2. 宽度策略
- 正文普通单图不应在视觉上接近正文阅读壳的心理满宽。
- 即使技术上仍受 `--heo-article-surface-max-width` 约束，也应通过额外宽度控制或容器内留白，使其明显属于正文流的一部分。
- 桌面端单图默认应比 gallery、更明显地保留左右呼吸感，避免形成“整块大画报”效果。

3. 高度策略
- 需要控制桌面端单图的最大展示高度，避免竖图、长图或高对比图片把正文首段直接压成“看图页”。
- 高度约束的目标不是压缩图片质量，而是避免图片先于正文建立主舞台关系。

4. 上下留白策略
- 图片前后留白应服务正文段落，而不是制造独立 section 感。
- 前后间距需要明显大于普通段落行距，但不能大到像在正文中强行插入一张海报区。
- 图片前后的文字与标题应保持阅读连续性，避免图片一出现就把正文切成两个孤立场景。

5. 圆角 / 边框 / 阴影策略
- 正文普通单图继续沿用当前正文媒体 token 体系，不重新引入与页头封面同级的重阴影、重叠光或强装饰边框。
- 单图的卡片感只用于建立嵌入块边界，不用于制造额外视觉中心。

6. caption 策略
- 凡存在图片说明文案，caption 应视为正文媒体块的一部分，用于补充语义并帮助图片降权回归阅读系统。
- caption 语气应弱于正文标题、强于装饰性注脚，避免 caption 缺席后单图只剩纯视觉冲击。

7. 与 gallery 的分级关系
- gallery 可以比普通单图更强调组合感与展示感，但其强度提升应来自“多图编排”而不是“更像主视觉海报”。
- 普通单图与 gallery 不应继续共享完全一致的视觉势能。

8. 与展示型大图的边界
- 若未来需要保留作品展示图、摄影图、海报图等强展示场景，应增加独立语义标识或独立 class，而不是继续覆盖普通单图规则。
- 默认正文单图规则只服务“普通阅读内容中的图片”，不负责承担所有大图展示诉求。

#### 16.8.3.6.4 正文 gallery 优化判断与原则
在正文媒体分级中，gallery 不应被继续理解为“几张图片并列摆放”，而应被视为正文中的局部展陈单元。它可以比普通单图保留更强一点的展示感，但本质上仍然服务正文叙事，而不是接管页面主视觉。

当前判断：
- `components/NotionPage.js` 已通过 `.notion-row.heo-image-gallery-row` 为正文 gallery 提供稳定语义入口，第一轮优化无需先补 DOM。
- `themes/heo/style.js` 已让 gallery 与普通单图产生初步分流，但当前“强”仍然较多依赖外层容器背景、圆角和阴影，而不是组内编排秩序。
- 后续真正值得优化的方向，不是继续把 gallery 做得更打，而是让它更像“被编辑过的一组图”，同时继续服从正文节奏。

后续所有正文 gallery，统一按以下原则判断：

1. 语义定位
- gallery 属于正文中的局部图组、并置内容或小型展陈，而不是第二封面、章节海报或作品集首页模块。
- gallery 的强化目标应是帮助多图建立关系感，而不是单纯制造更大的视觉音量。

2. 强度来源
- gallery 比普通单图更强的部分，应主要来自多图之间的组合关系、节奏和秩序，而不是更厚的卡片壳、更重的阴影或更明显的悬浮反馈。
- 如果一个 gallery 的第一眼存在感主要来自外层大容器，而不是内部图片编排，通常说明方向偏了。

3. 容器策略
- 外层容器只负责提供轻量级分组边界，不应在正文中形成过强卡片感。
- 可以保留适度背景、圆角和边框，但应优先减轻“大块组件”感，避免读者先看到容器，再看到图片关系。

4. 组内秩序策略
- 优先优化图片之间的间距、列宽、裁切逻辑、圆角统一性与上下节奏，让整组图先形成秩序感。
- gallery 的精致感应更多来自内部编排，而不是外壳装饰。

5. 与单图的关系
- 普通单图强调“嵌入正文流”，gallery 强调“多图共同构成一个内容单元”。
- 二者需要可感知分级，但不应走向两个完全割裂的视觉系统。

6. 与封面的关系
- 页头封面负责开场与情绪建立，gallery 只负责正文局部展示。
- 无论图组多美，gallery 都不应在页面中形成比封面更主动的主视觉竞争。

7. 数量分级预留
- 后续可按 2 张、3 张、4 张及以上做轻量分级，但前提是先把通用 gallery 的组内秩序和容器强度调稳。
- 不建议在尚未稳定基础规则前，就为所有数量情况同时做重编排。

#### 16.8.3.6.5 正文 gallery 后续实施计划
为避免后续直接进入样式细节争论，gallery 优化后续统一按以下阶段推进：

第一阶段：现状确认与问题收束
1. 盘点 `themes/heo/style.js` 中 gallery 容器、列布局、列内图片、caption 的现有规则，确认哪些视觉强度来自容器，哪些来自组内编排。
2. 对比普通单图当前效果，确认 gallery 的现有差异点是否已经足够表达“组图”身份，还是仍过度依赖外层卡片感。
3. 输出“保留什么、减弱什么、强化什么”的最小策略，避免再次靠截图感受临时试错。

第二阶段：最小可用优化
4. 优先减轻 gallery 外层容器的卡片化存在感，包括背景、边框、阴影与整体包裹感。
5. 同时增强组内秩序感，重点处理列间距、图片圆角、裁切一致性、列内上下节奏。
6. 若存在 caption，优先让 caption 与整组形成稳定归属，而不是继续只服务单张图说明。

第三阶段：轻分级与回归验收
7. 若通用 gallery 稳定，再评估是否按 2 张 / 3 张 / 4 张及以上建立轻量分级规则。
8. 回归检查长文中段、图文交替段落、暗色模式、专注阅读态与移动端压缩后的稳定性。
9. 最后再决定是否需要为“作品展示型图组”补独立语义，而不是继续挤占普通正文 gallery 的规则空间。

推荐优先修改的实现入口：
- `themes/heo/style.js`
- `components/NotionPage.js`（仅在后续确实需要补充分组语义时再评估）

验收补充：
1. gallery 是否仍然清楚地被感知为“一组相关图片”，而不是几张单图的机械并排。
2. gallery 的存在感是否更多来自内部秩序，而不是外层大卡片壳。
3. gallery 是否比普通单图略强，但仍明显弱于页头封面主视觉。
4. caption 存在时，是否已经更自然地归属整组图片，而不是漂浮在某一列之下。
5. 长文阅读过程中，gallery 是否提供了局部展陈感，而没有明显打断正文流。

#### 16.8.3.6.6 正文 gallery 优化 TODO 清单
为避免后续把 gallery 继续朝“更强展示组件”方向误推，正文 gallery 后续统一按以下 TODO 清单推进：

1. 复核 gallery 强度来源
- 区分当前视觉存在感究竟来自外层容器，还是来自组内图片编排。
- 明确哪些样式在帮助建立图组关系，哪些样式只是在增加卡片感。

2. 减轻外层容器卡片感
- 降低背景、边框、阴影与整体包裹感，让 gallery 更像正文局部展陈，而不是插入型大组件。
- 保留必要边界，但避免形成明显“第二模块”。

3. 强化组内图片秩序
- 统一列间距、列内节奏、圆角和裁切逻辑，让整组图先建立一致感。
- 优先解决“像几张图拼在一起”而不是“像一组图”的问题。

4. 明确 caption 的整组归属
- 如果存在图组说明文案，应优先服务整组语义，而不是只像某一张图的补注。
- 通过位置、间距、字级与对齐方式提升整组归属感。

5. 评估数量分级的必要性
- 在基础规则稳定后，再判断 2 张 / 3 张 / 4 张及以上是否需要轻量差异化处理。
- 避免过早引入复杂数量分流，导致规则维护成本失控。

6. 回归验收封面、单图与 gallery 主次
- 验证封面、普通单图、gallery 是否形成“封面最强、gallery 次之、单图更收”的稳定关系。
- 重点检查长文中段、图文交替、暗色模式、专注阅读态与移动端压缩效果。

## 17. 季节氛围特效补充规范

### 17.1 当前阶段判断
在首页首屏结构、卡片体系、移动端阅读节奏和头图干扰控制逐步收口后，当前 `HEO` 主题已经具备引入“轻量环境动效”的条件；但这一类特效只能作为氛围层，而不能重新成为新的视觉主角。

本轮结论确认：
- 季节特效适合 `HEO` 主题当前气质，但必须坚持“低干扰、低密度、低频率”。
- 特效的职责是补充季节感和页面情绪，不承担信息表达与品牌识别职责。
- 若特效开始影响阅读、抢占标题、干扰卡片边界或增加首屏压迫感，应优先减弱甚至关闭，而不是继续叠加新效果。

### 17.2 适用范围
- `themes/heo/components/AmbienceEffects.js`
- `themes/heo/index.js`
- `themes/heo/config.js`
- `themes/heo/style.js` 中与全局层级、特效层、暗色模式相关规则

### 17.3 季节映射结论
当前阶段最推荐的四季映射如下：
- 春：樱花 / 花瓣
- 夏：阳光 / 暖金色光斑 / 轻微热感粒子
- 秋：落叶
- 冬：流星 / 雪粒 / 冷色星点

补充判断：
- `冬天 -> 樱花` 不作为默认推荐方案。视觉上虽然浪漫，但季节语义不准确，容易形成“好看但不像冬天”的认知偏差。
- `春天` 与 `夏天` 不建议共用同一套“太阳”表达。春天更适合“花与风”，夏天更适合“光与热”。
- 若当前阶段只保留一套长期稳定特效，优先推荐：`浅色模式樱花 + 暗色模式低频流星`。

### 17.4 视觉定位规范
- 季节特效只能处于页面背景层或全局氛围层，不得进入正文内容层、标题层或交互层。
- 特效不应成为页面中最亮、最快、最密集的视觉对象。
- 特效应优先通过：
  - 透明度
  - 速度
  - 稀疏度
  - 色彩弱化
  来建立氛围，而不是靠数量堆叠存在感。
- 亮色模式下特效应更轻、更透；暗色模式下特效允许稍微增强，但仍要低于正文与主卡片的视觉强度。

### 17.5 动效强度约束
- 单屏可见粒子数量必须严格受控，优先少而稳，不追求“满屏在动”。
- 推荐方向：
  - 樱花：低密度、慢速、轻旋转、轻漂移
  - 流星：低频、短时出现、细线条、不连续刷屏
  - 落叶：数量少于樱花，速度略慢，面积略大但透明度更低
  - 夏季光斑：更接近环境光感或浮尘，不做实体图标感太阳
- 动效时长、轨迹和频率必须先服务“看不烦”，再考虑“看得见”。

### 17.6 阅读与交互保护原则
- 所有季节特效必须：
  - `pointer-events: none`
  - 不遮挡正文、导航、按钮、卡片主标题
  - 不破坏 `hover / focus-visible / active` 的状态识别
- 文章页、移动端和暗色模式必须单独验收，不能只在首页桌面端看效果。
- 若页面已经存在强封面图、玻璃卡、动画 Banner 或 Loading 过渡，季节特效应进一步减弱，避免形成“多个动势层叠加”。

### 17.7 移动端约束
- 移动端特效强度必须默认低于桌面端。
- 小屏优先减少：
  - 粒子数量
  - 动画轨迹长度
  - 闪烁与高光强度
- 移动端不得为了保留氛围而牺牲首屏可读性、卡片边界清晰度或标题识别。
- 若某一季节特效在移动端容易形成噪点感，应允许直接降级或关闭，而不是强行复刻桌面端体验。

### 17.8 可访问性与系统策略
- 必须尊重 `prefers-reduced-motion`，用户偏好减少动态效果时，应直接关闭季节特效。
- 特效总开关必须存在，且允许分模式独立控制。
- 季节特效不应依赖用户交互才能避开内容，也不应要求用户手动关闭后页面才变得可读。

### 17.9 配置落地建议
当前阶段建议至少保留以下配置能力：
- 总开关：是否启用季节氛围特效
- 浅色模式特效：如樱花 / 夏季光斑
- 暗色模式特效：如流星 / 冷色星点
- 后续若扩展为完整四季方案，再增加：
  - 自动按月份切换
  - 手动指定季节
  - 强度等级（低 / 中 / 高）
  - 移动端降级策略

当前已落地的轻量实现方向可作为第一阶段基线：
- `HEO_AMBIENCE_EFFECTS`
- `HEO_AMBIENCE_SAKURA`
- `HEO_AMBIENCE_METEOR`

### 17.10 当前阶段推荐方案
若以后要正式扩展成四季方案，当前最推荐的组合为：
- 春：樱花
- 夏：阳光光斑
- 秋：落叶
- 冬：流星

原因如下：
- 季节辨识度高
- 与 `HEO` 当前“阅读 + 氛围 + 克制表达”的方向兼容
- 不容易落入廉价模板特效感
- 便于在亮色 / 暗色模式下做自然分工

### 17.11 后续实施顺序建议
1. 先将现有 `浅色樱花 + 暗色流星` 打磨为稳定低干扰基线，并完成亮色 / 暗色 / 首页 / 文章页 / 移动端验收。
2. 再抽象“季节模式”配置结构，决定是否按月份自动切换。
3. 再补夏季光斑与秋季落叶，不直接一次性同时上四套。
4. 最后才评估冬季是否需要在流星之外增加极轻雪粒或冷色星点补充层。
