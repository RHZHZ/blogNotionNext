# HEO About 页面 UI 设计文档

## 1. 设计目标

本页不是单纯做静态展示，而是要在以下前提下长期稳定：

- 数据量会增长
- 各模块内容长度不一致
- 移动端与暗色模式都需要保持可读性
- About 页承担的是“概览 + 引导”职责，不是把所有信息全集强塞进首屏

核心原则：

1. 概览型内容可以并排
2. 可持续增长的长内容不应长期强行并排
3. About 页优先展示摘要，完整内容应支持独立展开、独立区块或跳转查看
4. 布局要先服务真实数据，再服务表面整齐

---

## 2. 信息架构定位

About 页建议分为两类内容：

### 2.1 概览型内容
适合卡片化、双栏化展示：

- Hero
- Personality / Hobby
- Skills
- Highlights
- Feature Cards
- Contacts

特征：

- 单项信息长度相对可控
- 可通过换行、换列、栅格吸收数据波动
- 适合首屏建立印象

### 2.2 增长型内容
不适合长期与短卡片强行并排：

- Books / Recent Books
- Timeline（当条目较多时）
- Notion 正文“更多介绍”

特征：

- 条目数可能持续增加
- 单项高度波动明显
- 容易造成一侧过高、另一侧大面积空白
- 更适合独立区块、摘要展示或跳转查看全部

---

## 3. 页面结构建议

推荐结构如下：

1. Hero
2. Persona + Skills
3. Focus + Feature Cards
4. 阅读与书架（独立区块）
5. 更多介绍 / Notion 正文（独立区块）

说明：

- `Persona + Skills` 可以保留双栏
- `Focus + Feature Cards` 可以保留双栏
- `Books + More Details` 不建议长期并排，应拆分为上下独立区块

---

## 4. 模块布局策略

## 4.1 Hero

定位：页面首屏概览。

要求：

- 标题、简介、状态信息保持清晰层次
- 不承载过多长列表内容
- 统计项数量宜少而精

不建议：

- 在 Hero 中堆叠太多标签或动态内容
- 让 Hero 成为二次滚动区域

---

## 4.2 Persona + Skills

定位：个性与能力概览。

结论：适合并排。

原因：

- 两侧都属于摘要型模块
- 适合通过栅格吸收数据增长
- 对页面第一屏的信息密度友好

设计要求：

- `skills` 使用自适应栅格
- 卡片允许高度自然增长，不做死板固定高
- 技能数量明显增加时，优先换列，不强制压缩
- 若后续技能特别多，可考虑“前 N 项 + 更多”

---

## 4.3 Focus + Feature Cards

定位：近期关注与补充说明。

结论：可以并排，但要控制左侧内容类型。

左侧适合放：

- tags
- contacts
- 少量重点信息

左侧不适合放：

- 长列表
- 长正文
- 持续增长的数据流

右侧 `featureCards`：

- 适合放阶段说明、方法论、当前重点
- 标签数量可多，但需自动换行

---

## 4.4 Timeline

定位：阶段经历说明。

结论：

- 条目较少时，可以放在右侧信息区
- 条目明显增加时，应单独抽为独立 section

判断标准：

- 如果时间轴长度明显高于旁侧卡片，不应继续强行并排
- About 页中优先展示摘要版时间轴，完整时间轴可独立展示

推荐策略：

- 默认展示最近 3~4 条
- 更多阶段可在独立区块完整展示

---

## 4.5 阅读与书架

定位：About 页中的特色摘要模块，不承担完整书库职责。

结论：不建议与“更多介绍”长期并排。

原因：

- 书籍数量会持续增加
- 标题、作者、summary 长短不一
- 最爱书籍与最近在读都可能增长
- 与正文并排时，极易形成高度失衡

设计原则：

- About 页展示“精选摘要”，不是“完整全集”
- 书籍卡片要限制摘要行数
- 列表数量要有首屏上限
- 完整书架应通过“查看更多 / 独立书架页”承接

推荐展示策略：

- 最爱书籍：展示前 4~6 本
- 最近在读：展示前 3~5 本
- 摘要限制行数，避免单卡无限拉长
- 保留跳转入口查看完整书架

不建议：

- 在 About 页直接无限展示全部书籍
- 给书架卡片设置一个很高的内部滚动容器作为长期方案
- 明知书单会持续增长，还继续与正文双栏并排

---

## 4.6 更多介绍 / Notion 正文

定位：完整信息承载区。

结论：应独占一行，按正常文档流自然增长。

原因：

- 正文长度天然不确定
- 可能包含标题、列表、图片、引用等复杂结构
- 与增长型列表并排会严重破坏页面平衡

设计要求：

- 独立 section
- 正常向下延展
- 不做内部滚动盒子
- 不与书架等增长型模块长期绑定双栏

---

## 5. 关于“无限增长列表”的统一原则

当某模块存在明显增长风险时，优先级如下：

1. 拆分为摘要展示 + 查看全部
2. 独立成单独区块
3. 分页 / 懒加载 / 折叠
4. 最后才考虑局部限高 + 内部滚动

说明：

- 内容展示型页面不应过度依赖内部滚动
- 内部滚动更适合侧栏、下拉面板、弹窗列表、工具型区域
- About 页这类内容页应优先保持单一主滚动

---

## 6. 并排与拆分的判断标准

看到一个模块时，先问两个问题：

### 6.1 这个模块会不会明显持续长高？
如果会，就不适合长期和短卡片并排。

### 6.2 这个模块是摘要型，还是完整承载型？

- 摘要型：可以卡片化、并排展示
- 完整承载型：应优先独立区块

---

## 7. 响应式策略

### 桌面端

- 概览区允许双栏
- 增长型内容优先单栏独立展示
- 避免为了对称感强行保持 1:1 双栏

### 平板端

- 双栏可以保留，但要开始偏向单栏收敛
- 对增长型模块要更谨慎

### 手机端

- 优先单栏
- 列表项数量控制更严格
- 避免单项内容过长导致首屏被单卡占满

---

## 8. 当前 About 页的最终结构结论

建议保留并排：

- `Persona + Skills`
- `Focus + Feature Cards`

建议条件并排：

- `Timeline` 仅在条目较少时保留在右侧信息区

建议拆分为独立区块：

- `AboutBooks`
- `更多介绍 / Notion 正文`

推荐顺序：

1. Hero
2. Persona + Skills
3. Focus + Feature Cards
4. 阅读与书架
5. 更多介绍

## 10. 卡片内部内容样式设计

### 10.1 信息层级原则

卡片内部应遵循固定的信息顺序，避免每张卡都像一篇小文章：

1. `eyebrow` / 分类标识
2. `title` / 核心信息
3. `accent` / 补充定位
4. `description` / 简述
5. `tags` / 状态 / 操作入口

设计原则：

- 一张卡片只允许一个视觉主标题
- 次级说明必须明显弱于标题
- 不同卡片之间层级规则保持一致
- 内容密度越高，越需要减少装饰性元素

### 10.2 文案长度控制

#### 标题

- 优先 1~2 行内读完
- 超长标题应截断或限制最大宽度
- 不应让标题撑坏整张卡的节奏

#### 描述

- 摘要型卡片建议控制在 2~4 行
- 卡片不是正文容器，不应承载长段落
- 需要完整表达时，应通过“更多介绍”或独立区块承接

#### 标签

- 标签用于快速识别，不用于承载完整句子
- 标签应该短、薄、可换行
- 标签数量过多时应优先折叠，而不是挤满卡片

#### 联系方式

- 长 URL / 邮箱必须允许换行
- 不应使用会破坏阅读节奏的强硬断词方式
- label 与 value 的视觉层级要稳定，不要两者一样重

### 10.3 不同卡片内容建议

#### Hero

当前内容包含：

- badge
- title
- subtitle
- description
- tagline
- introduction
- introductionMore
- avatar
- stats

判断：当前 Hero 信息量偏大。

问题：

- `subtitle`、`description`、`introduction`、`introductionMore` 都在承担介绍职责，语义重叠
- 首屏出现过多连续文本，会削弱标题与头像的聚焦力

建议：

- `Hero` 只保留一组核心介绍 + 一组补充说明
- `subtitle` 与 `description` 至少二选一压缩表达
- `introductionMore` 更适合下移到正文或单独 section
- `stats` 保持 3 项以内较合理

#### Persona / Hobby 卡片

建议：

- `eyebrow` 作为分类标签即可，不应过重
- `title + accent` 构成主信息组
- `description` 控制在 2~3 行内
- `badge` 只能是补充亮点，不应抢标题
- 图片应作为氛围元素，不应压过文字可读性

#### Skills 卡片

建议：

- 一项技能只保留：符号、名称、方向
- 避免加入过多解释文案
- 技能卡片要偏工具化、快速扫描化
- 当技能数较多时，优先保证可扫读，不追求每卡信息丰富

#### Focus / Feature Cards

建议：

- 左侧 `highlights` 做轻量标签流
- `contacts` 保持简洁，避免和 highlights 争抢视觉中心
- 右侧 feature card 的 `description` 不宜过长
- feature tags 应辅助理解，不应多到喧宾夺主

#### Timeline

建议：

- `year`、`title`、`description` 的对比要明确
- `description` 要比正文更短、更像阶段摘要
- 条目多时优先裁剪，而不是每条写得很满

#### Books

建议：

- 核心信息顺序：封面 > 标题 > 作者 > 摘要 > 状态
- 摘要应严格限制行数
- 不要在书卡中同时堆太多标签、注释、操作
- 书卡应服务“快速建立阅读印象”，而不是承载完整读书笔记

## 11. 卡片外观样式设计原则

### 11.1 卡片本体设计

卡片样式应服务内容分组，而不是成为视觉负担。

建议原则：

- 统一圆角体系
- 统一边框强度
- 统一阴影逻辑
- 统一内边距节奏
- 强调“分组感”，弱化“堆砌感”

当前 About 区最需要避免的是：

- 卡片套卡片的套娃感
- 外层遮罩与内层边框不贴合
- 某些卡片装饰过重、某些卡片又过轻，导致系统不统一

### 11.2 边框与圆角

建议：

- 所有主卡片使用统一圆角体系
- 遮罩层、渐变层、伪元素都必须继承卡片圆角
- 不允许出现外层矩形遮罩与内层圆角卡片冲突

### 11.3 阴影与层次

建议：

- 浅色模式用轻阴影 + 轻边框建立层次
- 暗色模式减少大面积浑浊阴影，更多依靠边框与亮度差建立层次
- 同层级卡片不应出现阴影强度明显失衡

### 11.4 内边距与呼吸感

建议：

- 标题区、正文区、标签区之间有稳定纵向间距
- 卡片内边距优先统一，再做个别特化
- 不要通过奇怪的额外 left padding 制造“伪层次”

### 11.5 装饰元素控制

建议：

- 渐变、光斑、遮罩可以有，但只能辅助氛围
- 装饰不能影响文字对比度
- 装饰不能制造假边框、假容器、假层级
- 一页中有 1~2 类装饰语言即可，不宜每块都不一样

---

## 12. 浅色 / 暗色模式设计原则

### 12.1 共同原则

无论浅色还是暗色，都应满足：

- 文本层级清晰
- 卡片边界可识别
- 图片不抢文本
- 标签、按钮、统计块有统一状态逻辑
- 不依赖单一颜色表达所有层级差异

### 12.2 浅色模式

目标：轻、清晰、分组明确。

建议：

- 背景以低噪声浅底为主
- 卡片用浅背景 + 轻边框 + 轻阴影
- 强调信息结构，不靠大面积颜色轰炸
- 彩色元素点到为止，优先用于 badge、标签、关键数字

避免：

- 卡片过白导致层次丢失
- 渐变过多导致页面花
- 标签和主标题竞争视觉焦点

### 12.3 暗色模式

目标：稳、透、可读，不发灰不发闷。

建议：

- 背景与卡片之间保持清晰亮度差
- 文本颜色分为主文字、次文字、弱提示三层
- 图片需要额外校正亮度和遮罩，保证可见但不刺眼
- 装饰层透明度在暗色模式下要更克制

避免：

- 整体发黑导致图片区与文本区糊成一片
- 用过重黑色遮罩压死背景图
- 用高亮纯白大面积直冲视觉
- 只靠阴影区分层次，结果暗部糊掉

### 12.4 关于图片与背景图

当前 About 页中最需要注意的是：

- `personality` 插画在暗色模式下不能太灰太暗
- `hobby` 背景图不能被深色蒙层彻底压死
- 背景图要服务氛围，不应削弱文字可读性

原则：

- 图片亮度适度提升
- 蒙层透明度按内容密度调节
- 文字区域始终优先于图像观感

## 13. 当前 About 区内容是否冗余

结论：当前内容存在一定冗余，主要问题不是“内容太多”，而是“介绍职责重复”。

### 13.1 Hero 冗余

当前 Hero 同时存在：

- `subtitle`
- `description`
- `introduction`
- `introductionMore`

这四组内容都在做“自我介绍”，语义有重叠。

建议：

- Hero 只保留一段短介绍 + 一段补充
- 多余说明下移到“更多介绍”
- 首屏以建立印象为主，不以完整表达为主

### 13.2 Feature Cards 与 Timeline 的信息边界略有模糊

当前这一区同时在讲：

- 近期关注
- 正在做的事
- 偏爱的表达方式
- 经历时间轴

问题：

- 都属于“我是谁 / 我在做什么 / 我如何做”的信息域
- 若文案继续增长，容易出现重复表达

建议：

- `Focus` 负责“当前关注点”
- `Feature Cards` 负责“方法论 / 长期倾向”
- `Timeline` 负责“阶段变化”
- 三者各自边界要更明确

### 13.3 Books 与 More Details 不应互相争夺主内容地位

问题：

- 书架是特色模块
- 正文是完整信息承载模块
- 两者都重要，但职责不同

建议：

- `Books` 做精选展示
- `More Details` 做完整承接
- 不要让两者在同一层级互相竞争页面中心

### 13.4 Skills 是否冗余

当前默认 `skills` 有 12 项。

判断：

- 作为能力画像不算冗余
- 但如果后续继续增长，需要避免把“会一点”都塞进去

建议：

- 保留“长期使用、能代表你”的技能
- 不要把技能区做成大而全技能词云

### 13.5 Contacts 是否冗余

当前联系方式 3 项，数量合理，不冗余。

但原则上：

- 联系方式应精简
- About 页不是导航页，不需要把所有外链全集都摆上来

---

## 14. 当前内容优化建议结论

建议保留：

- Hero
- Persona / Hobby
- Skills
- Highlights
- Feature Cards
- Books
- More Details

建议压缩：

- Hero 内重复介绍文案
- Feature 区过长说明
- 书卡摘要长度

建议按条件收缩：

- Timeline 条目数
- Skills 数量
- Highlights 数量
- Books 首屏展示数量

一句话补充结论：

> 当前 About 页不是模块太多，而是部分模块内部的信息职责还有重叠；优化重点应放在信息去重、层级收敛、摘要化展示，而不是简单删模块。

## 15. 各模块文案长度、标题长度、标签数量与截断规范

### 15.1 总体规则

About 页属于“概览页”，不是“全集页”，因此所有模块都应优先遵循以下原则：

- 首屏优先展示摘要，不展示无限全文
- 标题优先保证可扫读，不保证完整铺开
- 描述优先保证信息密度，不保证一次说尽
- 标签优先辅助识别，不承担主要叙述
- 超长内容优先下沉到独立区块、独立页面或 Notion 正文

### 15.2 截断处理优先级

当内容超出建议长度时，处理优先级如下：

1. 先判断是否属于冗余内容
2. 再判断是否应该下沉到下一级模块
3. 再决定是否截断
4. 只有摘要型模块才适合直接做视觉截断

说明：

- 标题可做单行或双行截断
- 摘要可做多行截断
- 联系方式值不建议视觉截断，优先换行
- 时间轴、正文、完整说明不建议纯视觉硬截断后不提供承接

---

### 15.3 Hero 模块

#### 建议字数

- `badge`：2~12 字
- `title`：2~16 字中文，或 1~4 个英文单词
- `subtitle`：12~28 字
- `description`：24~60 字
- `tagline`：8~24 字
- `introduction`：40~90 字
- `introductionMore`：40~100 字

#### 标题长度建议

- Hero 主标题应尽量控制在 1 行
- 若必须换行，最多 2 行
- 不建议使用特别长的句子型标题

#### 是否截断

- `title`：可限制 1~2 行，超出截断
- `subtitle`：建议最多 2 行，超出截断

### 18.9 一句话执行结论

先处理结构失衡，再处理信息冗余，最后处理视觉统一。

也就是说，本轮 About 改版最正确的推进方式不是“继续微调样式”，而是：

- 先拆结构
- 再删重复
- 再控数量
- 最后收样式

---

最终目标不是“每个模块都排得整齐”，而是：

- 数据多时不崩
- 内容长时不挤
- 移动端不失衡
- 暗色模式仍清晰
- 页面结构长期可维护

一句话总结：

> About 页默认承担“概览 + 引导”的角色，应优先展示摘要与印象；增长型内容应独立承接，而不是为了表面整齐长期强行并排。
