# HEO 移动端面板组件规范

## 1. 目的
将本次移动端抽屉菜单优化沉淀为可复用的项目内基线，用于后续统一 `HEO` 主题中的抽屉、侧滑面板、目录面板、搜索面板、筛选面板与移动端导航组件。

适用对象：
- `SlideOver`
- 移动端搜索面板
- 移动端侧边栏抽屉
- 目录抽屉 / TOC 面板
- 分类 / 标签筛选面板
- 其他移动端浮层式信息分组组件

---

## 2. 本次优化结论
本次抽屉菜单最终采用的不是“强装饰风格”，而是：

- 中性色基底
- 封面色弱参与
- 分区清晰但不过度描边
- 暗色模式降低边框与阴影存在感
- 当前态明确，但不做大面积高饱和铺色
- 标签、入口卡片、菜单项统一在同一视觉系统内

一句话总结：
**面板靠层级取胜，不靠强颜色取胜。**

---

## 3. 统一设计语言

### 3.1 结构层级
统一按以下顺序组织：
1. 面板容器
2. Hero / 顶部信息区
3. Section 标题
4. Section 内容容器
5. 卡片项 / 列表项 / 标签项
6. 当前态 / hover / 展开态

要求：
- 一个 section 只表达一个内容域
- 外层容器负责“分区”
- 内层卡片负责“交互项”
- 不允许“外层一圈 + 内层一圈 + 内阴影再一圈”同时很明显

### 3.2 色彩策略
- 基础背景优先使用深蓝灰 / 浅灰白中性色
- `accentColor` 只用于：
  - 当前态轻强调
  - 图标底色弱混合
  - 轻边框混色
  - badge
  - 箭头 / 焦点反馈
- 不允许把封面取色直接铺满整个面板或整个 section
- 暗色模式下，强调色必须进一步降饱和、降面积、降对比

### 3.3 暗色模式原则
- 暗色模式不是亮色反相
- 要主动压低：
  - border
  - shadow
  - overlay
  - 内发光
- 面板之间更多依赖背景层次区分，而不是明显描边区分
- 如果出现“套娃框”感觉，优先先减边框，再减阴影，最后再调背景透明度

### 3.4 状态设计
- 默认态：中性、稳定、可读
- hover：只允许轻微位移 / 轻微亮度变化
- active / current：
  - 可以提亮边框或图标
  - 可以增加极轻阴影
  - 不要与默认态拉开过大明暗差
- expand：展开态只表达结构变化，不额外制造视觉噪音

## 4. 组件级复用规范

### 4.1 面板容器
适用于抽屉、搜索面板、侧滑面板。

规范：
- 大圆角
- 轻渐变或轻玻璃感背景
- 暗色模式下保留实体感，不能过度透明
- 左右边缘强调只能保留一处，不要整圈强描边

### 4.2 Section 容器
适用于 `Navigation`、`Tags`、筛选分组、目录分组。

规范：
- section 是“信息分区”，不是二次主卡片
- 暗色模式下边框尽量更淡
- 优先用 padding、标题、背景明度做分区
- section 内部再放卡片时，section 本身不能太抢眼

### 4.3 菜单项 / 列表项
适用于导航、目录项、搜索结果项、快捷入口。

规范：
- 圆角卡片统一
- 图标、文字、箭头属于同一层级体系
- 当前态只比默认态略强一档
- 子菜单或二级项应通过缩进和更轻边界表达，不要再套一层厚卡片

### 4.4 Tag / Chip
适用于标签、分类筛选、快速过滤项。

规范：
- 必须支持 `flex-wrap + gap`
- 默认态为浅底 / 弱边框
- 选中态为强调色弱混合
- 数量角标应从属主标签，不可喧宾夺主

### 4.5 Hero / 头部区
适用于抽屉顶部说明块、搜索面板介绍块、目录概览块。

规范：
- 仅保留一个视觉中心
- badge 面积小、用途明确
- 关闭按钮或次级操作要弱于标题信息
- Hero 可比 section 稍强，但不能强过页面主体内容

## 5. 代码落地约定

### 5.1 推荐透传参数
以下参数应优先标准化：
- `accentColor`
- `isDarkMode`
- `className`
- `isActive`
- `variant`（如有必要）

### 5.2 推荐拆分方向
后续可逐步抽离公共样式语义：
- panel style
- section style
- item card style
- chip style
- icon capsule style

目标不是立即抽象过度，而是先统一语义，再择机复用。

### 5.3 开发顺序
新组件或旧组件改造时，按以下顺序执行：
1. 结构分区
2. 间距统一
3. 默认态
4. 当前态 / hover / 展开态
5. 暗色模式
6. accentColor 弱参与
7. 动效收尾

### 6.1 本轮已验证的具体结论
- 移动端目录应采用独立 `drawer` 变体，而不是直接复用桌面目录样式
- `showHeader={false}` 适合用于避免面板内外双标题重复
- 右栏“最新发布”更适合采用“纯列表文字 + 小缩略图角标”而非密集小卡片堆叠
- 暗色模式当前态更适合柔和暖橘高亮，而不是高饱和亮黄铺底
- 强调色应主要落在边框、文字与弱背景混色，不宜做大面积重铺色
- 顶部操作区不能只改单个按钮，还要一起处理外层胶囊容器与桌面 / 移动双态
- `ReadingProgress` 不应再以孤立黑色 badge 存在，而应并入顶部操作组语言

### 6.3 本轮新增：顶部导航与桌面端文章页标题胶囊
本轮顶部导航优化虽然不完全属于移动端面板，但已经形成了可复用的同源视觉基线，可作为 `HEO` 顶部导航和浮层类组件的参考：

- 顶部导航采用三栏结构，左右区域不再互相牵动。
- 中央内容区允许在“菜单组”和“文章标题胶囊”之间切换，但同一时刻只保留一个主视觉中心。
- 文章页中央文案改为直接显示文章标题，以增强内容语义与页面定位。
- 中央标题胶囊采用与左右区一致的胶囊节奏：大圆角、弱边框、柔和阴影、磨砂背景、轻内高光。
- 标题字色最终不再走偏透明灰路线，而是回收到与左右导航一致的深灰体系，避免在浅色磨砂底上出现“灰蒙”观感。
- 顶部导航项与下拉菜单项共享同一套亮/暗色交互态，并补齐 `focus-visible` 可访问性反馈。
- 下拉菜单触发器与菜单面板之间增加悬停桥接区，避免 hover 断层导致菜单过早收起。

#### 对移动端/面板系统的可复用启发
- “定位容器”和“视觉容器”应分离，避免一个 class 同时承担布局、动效和玻璃视觉，导致后续难以覆盖。
- 交互态设计应优先统一到 token 层，而不是在不同组件中零散叠加高优先级覆盖。
- 标题胶囊、菜单胶囊、操作组胶囊应共享统一的高度、圆角和阴影节奏，再根据内容密度做局部微调。
- 玻璃态文字必须先保证清晰，再做柔化；当背景已经有高亮磨砂时，文字不应再叠加明显白色高光阴影。
- 下拉面板容器本身也应进入 token 体系，避免菜单项已经统一但面板容器仍然散落硬编码。

#### 下一步开发计划（导航相关）
1. 做一轮桌面端 / 移动端 / 普通页 / 文章页 / 亮色 / 暗色联动回归，确认导航 token 化后没有局部漂移。
2. 重点检查长标题文章、带下拉导航项、`ReadingProgress` 出现/隐藏时的整体节奏是否稳定。
3. 继续细化顶部导航 token 语义，区分容器 token、标题 token、交互 token、submenu token。
4. 条件允许时补充验收截图基线，方便后续 UI 微调时快速对照。
5. 回归稳定后，再考虑把这套导航 token 下沉复用于移动端抽屉、搜索面板与浮层类组件。

## 7. 验收清单
每次改造后至少检查：
- 亮色模式是否稳定
- 暗色模式是否有“重叠框 / 套娃框”问题
- 当前态是否清晰但不过重
- 标签 / 筛选项是否会拥挤
- 小屏下是否存在横向挤压
- 图标、文字、箭头是否同属一个视觉系统
- accentColor 是否只是点缀而非污染背景

---

## 8. 项目执行原则
后续移动端面板类组件改造，统一遵循：

**先收敛，再增强；先统一，再个性化。**

如无明确需求，不新增新的高饱和视觉体系，不引入新的重描边卡片语言。
